# FunctionApp-and-Storage-account-using-private-endpoint

Below is a practical **use case + step-by-step build table** for a **Function App using inbound private endpoint + outbound VNet integration**, where the Function App connects to **Azure Storage through a storage private endpoint**.

### Example use case

Internal users or apps call the Function App over a **private endpoint**, and the Function App reaches its required **Azure Storage account privately** through **VNet integration + storage private endpoints**. In Azure, the Function App’s **private endpoint is only for inbound traffic**, while **VNet integration is for outbound traffic**. The VNet integration subnet must be **dedicated/empty** and delegated to **`Microsoft.Web/serverFarms`**; it cannot be the same subnet as the Function App private endpoint. ([Microsoft Learn][1])

## Detailed steps

| Step | Area                                               | What to do                                                                                                                                                                                                                                             | Key configuration / setting                                                                                          | Why it matters                                                                                                                                                                                                                                               |
| ---- | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 1    | Plan the topology                                  | Use a hosting plan that supports networking, such as **Elastic Premium** or a supported Dedicated/App Service tier.                                                                                                                                    | Function App on **Elastic Premium** is commonly used for private networking scenarios.                               | Azure Functions networking capabilities depend on hosting option; Premium supports VNet integration and private endpoints. ([Microsoft Learn][2])                                                                                                            |
| 2    | Create the VNet                                    | Create one VNet with at least **three subnets**: one for **Function inbound private endpoint**, one for **Function outbound VNet integration**, and one for **Storage private endpoint**.                                                              | Example: `snet-func-pe`, `snet-func-int`, `snet-stg-pe`                                                              | Separating these roles avoids subnet conflicts and follows Azure networking requirements. ([Microsoft Learn][1])                                                                                                                                             |
| 3    | Prepare the VNet integration subnet                | Create the **integration subnet** as an **empty dedicated subnet**. Delegate it to **`Microsoft.Web/serverFarms`**.                                                                                                                                    | Delegation: `Microsoft.Web/serverFarms`                                                                              | App Service/Functions VNet integration uses a delegated subnet. Azure also notes the subnet must be empty. ([Microsoft Learn][3])                                                                                                                            |
| 4    | Size the integration subnet correctly              | Give the integration subnet enough IP space.                                                                                                                                                                                                           | Minimum supported block shown as `/28`; recommended minimum is `/26` for growth.                                     | App Service consumes IPs from the integration subnet per instance and during scale operations. ([Microsoft Learn][3])                                                                                                                                        |
| 5    | Prepare the Function private endpoint subnet       | Create a separate subnet for the **Function App private endpoint**. Do **not** use the VNet integration subnet.                                                                                                                                        | Example: `snet-func-pe`                                                                                              | Azure states the VNet integration feature can’t use the same subnet as the private endpoint. ([Microsoft Learn][1])                                                                                                                                          |
| 6    | Prepare the Storage private endpoint subnet        | Create another subnet for the **Storage account private endpoint(s)**.                                                                                                                                                                                 | Example: `snet-stg-pe`                                                                                               | Keeping storage PE separate makes DNS and routing cleaner and avoids delegation conflicts. A delegated subnet can’t be used with a private endpoint. ([Microsoft Learn][4])                                                                                  |
| 7    | Configure private endpoint subnet policy           | On private endpoint subnets, disable **private endpoint network policies** where required.                                                                                                                                                             | Subnet setting: **Private endpoint network policies = Disabled**                                                     | This is the standard Azure setup for PE subnets. ([Microsoft Learn][5])                                                                                                                                                                                      |
| 8    | Create the Storage account                         | Create the Storage account used by the Function App. Prefer GPv2.                                                                                                                                                                                      | Storage account for host/runtime/content                                                                             | Private endpoints for storage aren’t available for GPv1 accounts. ([Microsoft Learn][6])                                                                                                                                                                     |
| 9    | Add Storage private endpoints                      | Create private endpoints for the storage services your Function App needs. At minimum, most Function scenarios need **Blob**; Premium/content-share scenarios often also need **File**. Queue/Table may also be needed depending on triggers/bindings. | PE subresources such as `blob`, `file`, `queue`, `table` as needed                                                   | Azure Storage private endpoints are created per storage service, and traffic stays on Microsoft backbone/private link. ([Microsoft Learn][6])                                                                                                                |
| 10   | Configure private DNS for Storage                  | Link the appropriate **Private DNS zones** to the VNet so the Function App resolves the storage account to its private IPs.                                                                                                                            | Typical zones include `privatelink.blob.core.windows.net`, `privatelink.file.core.windows.net`, and others as needed | Without DNS, the app may still resolve public endpoints and fail after public access is restricted. Azure states private endpoint scenarios require proper private DNS resolution. ([Microsoft Learn][6])                                                    |
| 11   | Restrict public access to Storage                  | After testing private resolution, restrict the storage firewall/public network access as required by policy.                                                                                                                                           | Public network access disabled or limited                                                                            | Azure notes private link creation alone does **not** automatically block public endpoint access; you must restrict it separately. ([Microsoft Learn][6])                                                                                                     |
| 12   | Create the Function App                            | Create the Function App on Premium or supported plan.                                                                                                                                                                                                  | During creation or after creation                                                                                    | This is the app that will have inbound PE + outbound VNet integration. ([Microsoft Learn][2])                                                                                                                                                                |
| 13   | Enable VNet integration                            | Integrate the Function App with the **delegated integration subnet**.                                                                                                                                                                                  | Function App > Networking > VNet Integration                                                                         | VNet integration enables **outbound** access from the app into the VNet and to private endpoints. ([Microsoft Learn][7])                                                                                                                                     |
| 14   | Enable route-all for outbound traffic              | Turn on **Route All** so all outbound traffic goes through the VNet. Prefer the site property **`vnetRouteAllEnabled=true`**. Legacy equivalent: **`WEBSITE_VNET_ROUTE_ALL=1`**.                                                                       | `vnetRouteAllEnabled=true` or `WEBSITE_VNET_ROUTE_ALL=1`                                                             | This is the main “update needed for the environment variable to use VNet” when you want the app’s outbound traffic to follow VNet routing. Azure marks `WEBSITE_VNET_ROUTE_ALL` as legacy and says `vnetRouteAllEnabled` replaces it. ([Microsoft Learn][8]) |
| 15   | Add inbound private endpoint for Function App      | Create a **private endpoint** for the Function App itself in the Function PE subnet.                                                                                                                                                                   | Private endpoint to the app (`sites`)                                                                                | This makes the app reachable privately; PE is **inbound only**. Outbound still uses VNet integration, not the PE. ([Microsoft Learn][1])                                                                                                                     |
| 16   | Configure private DNS for Function App             | Create/link the private DNS zone **`privatelink.azurewebsites.net`** and let Azure DNS zone group create records.                                                                                                                                      | Zone: `privatelink.azurewebsites.net`                                                                                | Required so internal callers resolve `myapp.azurewebsites.net` to the private endpoint IP. Azure also notes SCM/Kudu needs the corresponding private DNS handling. ([Microsoft Learn][1])                                                                    |
| 17   | Disable public access to Function App              | After validation, disable public network access or otherwise lock down public ingress.                                                                                                                                                                 | Function App public access off / access restrictions                                                                 | This ensures the app is only reachable from private networks through Private Link. ([Microsoft Learn][1])                                                                                                                                                    |
| 18   | Update storage-related app settings                | Keep or set the runtime/content settings used by Functions for storage.                                                                                                                                                                                | `WEBSITE_CONTENTAZUREFILECONNECTIONSTRING`, `WEBSITE_CONTENTSHARE` where applicable                                  | Azure warns changing/removing these can stop startup. They’re especially relevant on Premium/content-share scenarios. ([Microsoft Learn][8])                                                                                                                 |
| 19   | Enable content-over-VNet for Premium content share | If your Function App content share is in a storage account restricted to the VNet, enable **`vnetContentShareEnabled=true`**. Legacy equivalent: **`WEBSITE_CONTENTOVERVNET=1`**.                                                                      | `vnetContentShareEnabled=true` or `WEBSITE_CONTENTOVERVNET=1`                                                        | This is required for proper cross-stamp scaling when the content share storage is VNet-restricted in Premium/Dedicated plans. ([Microsoft Learn][8])                                                                                                         |
| 20   | Set DNS app setting if needed                      | If private DNS resolution is not being inherited/handled properly, add **`WEBSITE_DNS_SERVER=168.63.129.16`**.                                                                                                                                         | `WEBSITE_DNS_SERVER=168.63.129.16`                                                                                   | Azure notes this app setting is often required when using Azure Private DNS zones and private endpoints. ([Microsoft Learn][8])                                                                                                                              |
| 21   | Validate name resolution                           | From the Function App console or test code, verify the storage FQDN resolves to a **private IP** and the Function App FQDN resolves privately from inside the network.                                                                                 | DNS test / connection test                                                                                           | A private endpoint without working DNS is one of the most common failure points. Azure private endpoint guidance depends on private DNS mapping. ([Microsoft Learn][1])                                                                                      |
| 22   | Validate traffic flow                              | Test both directions: internal client → Function App private endpoint, and Function App → Storage private endpoint.                                                                                                                                    | Test HTTP/trigger behavior and storage access                                                                        | Confirms inbound PE and outbound VNet integration are both working as designed. ([Microsoft Learn][1])                                                                                                                                                       |
| 23   | Add NSG/UDR if needed                              | Apply NSGs and route tables to the **integration subnet** if you need outbound filtering or custom routing.                                                                                                                                            | NSG / UDR on integration subnet                                                                                      | Azure states VNet integration supports NSGs, UDRs, and NAT Gateway for outbound control. ([Microsoft Learn][7])                                                                                                                                              |
| 24   | Monitor and troubleshoot                           | Use App Insights, Log Analytics, storage network logs, and Function console logs to confirm startup, DNS, and storage mounting behavior.                                                                                                               | Check startup logs, DNS failures, mount errors                                                                       | Misconfigured DNS, wrong subnet usage, or missing content-share settings typically show up here first. Azure also notes app setting changes restart the app. ([Microsoft Learn][8])                                                                          |

## Environment variable / site setting updates you most likely need

| Setting                                    | Recommended value         | When to use it                                                                             | Notes                                                                                                                                             |
| ------------------------------------------ | ------------------------- | ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| `vnetRouteAllEnabled`                      | `true`                    | When all outbound traffic from the Function App should traverse the VNet                   | Preferred modern site setting. Replaces `WEBSITE_VNET_ROUTE_ALL`. ([Microsoft Learn][8])                                                          |
| `WEBSITE_VNET_ROUTE_ALL`                   | `1`                       | Legacy equivalent of route-all                                                             | Still seen in many deployments, but Azure marks it as legacy. ([Microsoft Learn][8])                                                              |
| `vnetContentShareEnabled`                  | `true`                    | When the Function content share is in a VNet-restricted storage account                    | Important for Premium/Dedicated scaling with secured storage. ([Microsoft Learn][8])                                                              |
| `WEBSITE_CONTENTOVERVNET`                  | `1`                       | Legacy equivalent of `vnetContentShareEnabled`                                             | Legacy, but still common in older templates. ([Microsoft Learn][8])                                                                               |
| `WEBSITE_DNS_SERVER`                       | `168.63.129.16`           | When using Azure Private DNS zones/private endpoints and DNS resolution needs to be forced | Often needed in private endpoint scenarios. ([Microsoft Learn][8])                                                                                |
| `WEBSITE_CONTENTAZUREFILECONNECTIONSTRING` | storage connection string | If the Functions content share is used                                                     | Required for Consumption and Elastic Premium; changing it can break startup. Azure Files MI support is not available here. ([Microsoft Learn][8]) |
| `WEBSITE_CONTENTSHARE`                     | unique share name         | When deploying to secured/VNet-restricted storage and especially via IaC                   | For secured storage, pre-create the share and use a unique name. ([Microsoft Learn][8])                                                           |

## Important design notes

* **Inbound vs outbound:**
  The **Function App private endpoint** is for **incoming** private access only. **Outbound** connectivity to storage/private resources uses **VNet integration**. ([Microsoft Learn][1])
* **Subnet separation:**
  Do not place **VNet integration** and **private endpoints** in the same subnet. The VNet integration subnet must be delegated to **`Microsoft.Web/serverFarms`**, while delegated subnets can’t be used for private endpoints. ([Microsoft Learn][1])
* **DNS is critical:**
  Private endpoints work only when the Function App resolves service names to **private IPs** through Private DNS or equivalent DNS forwarding. ([Microsoft Learn][1])
* **Storage firewall:**
  Creating a storage private endpoint does **not** automatically disable public access. Restrict the storage public endpoint separately. ([Microsoft Learn][6])
